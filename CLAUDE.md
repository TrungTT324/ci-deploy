# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

CI-Deploy is a **native Android Kotlin, multi-module Gradle** project — there is no Flutter/React Native anywhere in the repo (an explicit constraint left over from an earlier Flutter → native migration; never introduce `pubspec.yaml`, `lib/`, or run `flutter` commands). Three installable apps share a set of `:libs:*` feature modules:

- **`:app`** (`hdisoft.app.cideploy`) — the main CI-Deploy tool: a WebView portal client plus developer-tool features (LAN host discovery, Bluetooth chat/HID-mouse test, TCP test, API explorer, live logcat streaming, local web server), all wired up with OTA self-update.
- **`:appMouse`** (`hdisoft.app.mouse`) — a standalone "phone as Bluetooth HID mouse" app with its own APK, reusing `:libs:bluetooth` / `:libs:appupdate` / `:libs:core`.
- **`:appQa`** (`hdisoft.app.qa`) — a no-root QA automation tool (Accessibility-based gesture/text injection, floating overlay control, screen capture/record) that publishes its reports over `:libs:webserver`; see `appQa/README.md`.

Two non-Android Python side-projects sit alongside the Gradle project and are **not** part of the Gradle build: `agent-deploy/` (FastAPI dashboard + script to build/publish local debug builds) and `logcat-server/` (websocket log relay consumed by the `:libs:logcat` feature).

## Commands

### Android (Gradle, run from repo root)
- `./gradlew assembleDebug` — build `:app` debug APK → `app/build/outputs/apk/debug/app-debug.apk`. This is also the CI build (`.github/workflows/build-release.yml`, triggered on push to `main`/tags), which republishes it as `CI-Deploy_debug.apk` on a GitHub Release.
- `./gradlew :appMouse:assembleDebug` — build the AppMouse APK.
- `./gradlew :libs:<name>:compileDebugKotlin` or `:app:compileDebugKotlin` — fast compile-check of a single module without a full assemble; the quickest way to verify a change to one module.
- There are no test sources in the repo (`test`/`androidTest` dirs are empty) and no lint config — compiling is the only automated verification available.
- `./build.sh` — builds `:app` debug and publishes it plus `ci-deploy-version.json` to the local nginx dir (`/opt/homebrew/var/www/ci-deploy`), for OTA testing against a real device on the LAN.
- `./build_local.sh` — same flow for `:appMouse` (publishes to `/opt/homebrew/var/www/ci-deploy/apps`, writes `appmouse-version.json`, and also updates the portal's `app.json`).

### Python side-tools
- `agent-deploy/`: `./run.sh` (first run creates `.venv` and installs `requirements.txt`) starts a FastAPI build dashboard at `http://127.0.0.1:8000`; `python3 deploy.py` runs the single-project build+publish flow directly without the UI.
- `logcat-server/`: `./run.sh` starts the websocket log relay server used by `:libs:logcat`.

## Architecture

### Module layout & the standard per-module skeleton
`settings.gradle.kts` wires: `:app`, `:appMouse`, and `:libs:{appupdate,core,webserver,logcat,cidata,bluetooth}`. Every `:libs:*` module — and every `features/<name>` package inside `:app` — follows the same shape; copy this pattern when adding a new feature or module:
- `domain/model`, `domain/repository` (interface), `domain/usecase` (one class per operation, exposed as `operator fun invoke(...)`)
- `data/datasource`, `data/repository` (implementation of the domain interface)
- a `XxxServiceLocator` object (plain `by lazy` singletons — no Hilt/Dagger/Koin anywhere) acting as that module's own tiny DI container

`:app`'s top-level `ServiceLocator` (`app/src/main/java/hdisoft/app/cideploy/di/ServiceLocator.kt`) composes these per-module locators and exposes what `:app`'s ViewModels/Activities need; it's initialized once from `CiDeployApplication.onCreate()`.

### `:libs:bluetooth` is not a normal module — know this before touching Bluetooth code
It has no `src/` of its own. Its `build.gradle.kts` points its Kotlin source set directly at `:app`'s tree (`android.sourceSets.getByName("main").java.srcDirs("../../app/src/main/java/hdisoft/app/cideploy/features/bluetooth/data", ".../security/data")`), and `app/build.gradle.kts` in turn excludes those same paths from its own compilation (`sourceSets["main"].java.exclude(...)`). So `BluetoothConnector.kt`, `BluetoothHidMouseConnector.kt`, etc. under `app/src/main/java/hdisoft/app/cideploy/features/bluetooth/{data,security/data}/` are the **only copy** of that code — physically living in `:app`'s source tree under the `hdisoft.app.cideploy.features.bluetooth.data` package, but compiled into *both* `:app` directly and `:libs:bluetooth`'s artifact (via the borrowed source dirs). That's how `:appMouse` gets `BluetoothHidMouseConnector` without depending on `:app`. Editing those files affects both apps simultaneously — there is nothing to keep "in sync".

### OTA self-update (`:libs:appupdate`)
The single integration point is `AppUpdateChecker` (package `hdisoft.app.appupdate`; full contract in `libs/appupdate/README.md`). An integrating app constructs it with just a `Context` (+ optional `jsonFileName` override) and calls `checkUpdate(baseUrl)`; the JSON file name defaults to `<app-label-slug>-version.json`, derived from the app's own `android:label`/`app_name` (`"CI-Deploy"` → `ci-deploy-version.json`, `"AppMouse"` → `appmouse-version.json`), matching exactly what `build.sh`/`build_local.sh` publish.

The module owns the **whole** OTA lifecycle, not just check+download: `downloadApk(url, file)` → `Flow<DownloadState>`, then `installDownloadedApk(context, info, file, autoInstallIfRooted)` → `Flow<InstallState>` (`Verifying` → `Ready`/`Installing` → `Success`/`Error`) does verify (size/sha256/signing-cert match) and install. `autoInstallIfRooted` defaults to `DeviceUtils.isDeviceRooted()` (`:libs:core`) — when true, install happens silently via `su`/`pm install` (`AppUpdateInstaller.installApkRootSilent`) with no user interaction; when false, the flow stops at `InstallState.Ready` and the app calls `installApk(context, file)` on user confirmation, which requests the "install unknown apps" permission itself if missing before opening the system installer. `AppUpdateInstaller.kt` holds this verify/permission/install implementation; call it directly only for finer control than `installDownloadedApk` gives.

Root *detection* itself (`DeviceUtils.isDeviceRooted()`) deliberately stayed in `:libs:core` rather than moving into `:libs:appupdate` — `:libs:logcat` and `TestTcpActivity` depend on it too, unrelated to OTA. It used to live in a dedicated `FileUtils` in the same package; that class no longer exists — it's folded into `DeviceUtils` alongside `isEmulator()`/`getDeviceId()`, since "is this device rooted" is squarely a device-info question. Everything else that used to live in the old `FileUtils` (`verifyUpdateApk`, `installApk`, `installApkRootSilent`, `cleanupTempApkFiles`) had already moved into `:libs:appupdate`'s `AppUpdateInstaller` earlier, since that was 100% update-specific; `:app`'s `MainViewModel`/`MainActivity` now just collect `installDownloadedApk`'s `Flow<InstallState>` instead of calling install/verify functions directly — this is also what backs `:app`'s silent-root-install flow used in the rooted-emulator test setup documented in `device_rooted.md`.

`isDeviceRooted()` caches a **positive** result forever but never caches a negative one — a `su` check can legitimately fail on the first-ever call (root daemon not warmed up right after boot, or the root manager's own grant prompt not yet answered) and succeed later; caching that early `false` would otherwise permanently disable the whole silent-install path for the rest of the process even once root becomes usable. Don't "simplify" this back to caching both outcomes.

It also tries **two different `su` invocation syntaxes**, not just one — verified directly against a real Google APIs (non-Play Store, non-Magisk) emulator: its bundled AOSP/toybox `su` has no `-c` flag at all (usage is `su WHO COMMAND...`), so `su -c "id"` fails with `invalid uid/gid '-c'` even on a genuinely rooted device (`adb root` / `adb shell id` both confirm `uid=0`). `isDeviceRooted()` now tries Magisk/SuperSU-style (`su -c "<command>"`) first, then falls back to `su 0 sh -c "<command>"` (toybox-compatible). `AppUpdateInstaller.installApkRootSilent` was **not** updated to try both — it still assumes Magisk-style `-c` only, so root-install can still fail on a toybox-`su` device even once detection correctly reports rooted; that's a known follow-up, not yet requested. Full flow guarantee (rooted = zero confirmation UI end-to-end, except a genuine root-install *failure* which intentionally still shows a manual-install fallback dialog) is in `docs/appupdate-flow.md`.

The dialog UI for this whole flow (upgrade prompt → download progress → Ready/manual-install) also moved into the module, as `AppUpdateDownloadFlow` — it used to be ~250 lines split across `MainActivity`'s `showUpgradeDialog`/`setupAndStartDownload`/`handleDownloadState`/`handleInstallState`/`showManualInstallDialog`/`showManualInstallUI`/`showReadyInstallDialog`. It deliberately does **not** own the download/install state itself (that stays in `MainViewModel`'s `downloadState`/`installState` StateFlow, so it survives Activity recreation on rotation) — it's fed via `onDownloadStateChanged`/`onInstallStateChanged` from the same `Flow.collect` sites that used to call the old handler functions, and calls back into the ViewModel through a small `AppUpdateDownloadActions` adapter `:app`'s `MainActivity` implements inline. One instance per Activity instance (it holds live dialog references). `:app`'s `MainActivity` still owns resolving the actual `downloadUrl` from `UpdateInfo.url` (host/urlPath rewriting is CI-Deploy-specific) before calling `onUpdateAvailable`.

Two settings live in `AppUpdateSettings` rather than each app rolling its own: the periodic-check-interval (`getCheckIntervalSeconds`/`setCheckIntervalSeconds`, options `CHECK_INTERVAL_OPTIONS`) and the update source (`getUpdateSourceMode`/`setUpdateSourceMode`, enum `UpdateSourceMode.LAN_ONLY`/`PUBLIC_HOST`). Actually *using* them — driving a periodic timer, or picking which `baseUrl` to pass into `checkUpdate()` — is still each app's own responsibility (`:app`'s `MainViewModel.checkForUpdates` picks `"http://$host:8080/ci-deploy"` for `LAN_ONLY` vs `AppUpdateSettings.DEFAULT_PUBLIC_BASE_URL` for `PUBLIC_HOST`). **`DEFAULT_PUBLIC_BASE_URL` is currently a placeholder** (`https://TODO-public-host.example.com/ci-deploy`) — `PUBLIC_HOST` mode silently never finds an update until that constant is replaced with a real publicly-reachable URL.

`AppUpdateSettings` also tracks read-only history — `getLastCheckedAtMillis()` and `getLastApkFilePath()` — written automatically by `AppUpdateChecker` as a side effect of every `checkUpdate()`/successful `downloadApk()` call (the setters are `internal`, so only the module itself writes them).

A ready-made settings UI over all of the above ships in the module too: `AppUpdateInfoDialog.show(context, versionName, buildNo, onIntervalSaved, onSourceModeSaved)` (`:app`'s `MainActivity` calls it with its own `BuildConfig.VERSION_NAME`/`BuildConfig.BUILD_NO`, since those are app-specific and can't be read from inside the module). It used to live in `:app` as a local `InfoDialog`; moved into `:libs:appupdate` so `:appMouse` (or any future app) can reuse the same dialog instead of rebuilding it. Its text comes from the module's own `res/values/strings.xml` (English, default) and `res/values-vi/strings.xml` (Vietnamese) — resolved via `context.getString(...)` against whatever `Context` the integrating app passes in, so it automatically matches that app's current locale with no locale-picking code. `:libs:appupdate` didn't have a `res/` directory before this.

### Shared SharedPreferences wrapper (`:libs:core`)
Several modules used to hand-roll their own `context.getSharedPreferences(name, MODE_PRIVATE)` + `.edit().put...().apply()` class (`AuthLocalDataSource`, `WebConfig`, `ApiSessionPrefs`, the Bluetooth `*Store` classes, etc. still do). New prefs-backed code should instead use `hdisoft.app.core.prefs.PrefsStore(context, name)` (typed get/put for String/Int/Long/Boolean, plus a batched `edit { ... }`) — `AppUpdateSettings`, `AppUpdateInfoDialog`, and the moved-in `HostLocalDataSource` (below) are the current examples. A module needs `implementation(project(":libs:core"))` to reach it (already true for `:app` and most `:libs:*`; `:libs:appupdate` had it added specifically for this).

Both apps' `BUILD_NO` (compared against the server JSON's `buildNo`) is generated at Gradle configuration time from the current timestamp (`buildConfigField("long", "BUILD_NO", ...)` in each app's `build.gradle.kts`) and printed to the build log as `CI-DEPLOY BUILD_NO GENERATED: <n>` / `APPMOUSE BUILD_NO GENERATED: <n>`. `build.sh`, `build_local.sh`, and `agent-deploy/deploy.py` all scrape that exact log line to stamp the published version JSON — preserve the log format if you touch the build number generation.

`:app`'s debug and release build types share one signing key (`app/ci-deploy.jks`, configured in `app/build.gradle.kts`) intentionally: Android only allows an OTA install to replace an already-installed APK when certificates match, and OTA always ships the debug artifact.

### Host discovery (`hdisoft.app.appupdate.hostdiscovery.*`, inside `:libs:appupdate`; used only by `:app`)
Used to be its own module (`:libs:hostdiscovery`); moved bodily into `:libs:appupdate` (same clean-architecture shape, just repackaged) so it could be gated by the module's own `AppUpdateSettings.UpdateSourceMode`. `HostDiscoveryService.startDiscovery()` **always tries the last known host first** (a real HTTP reachability check via `HostRemoteDataSource.verifyHost`), regardless of the source mode — only *after* that fails does the mode matter: `LAN_ONLY` falls back to sweeping `<subnet>1`–`<subnet>254` on port 8080 (`HostRemoteDataSource.discoverHost`, `Semaphore(80)`, first success wins via a conflated `Channel` that cancels the rest, up to 3 retries), `PUBLIC_HOST` reports `State.Error` immediately (nothing else to try — no `State.Disabled` anymore, that got folded into `Error` since by that point a real failure already happened). `startDiscovery()` is a no-op while already `State.Progress`, so it's safe to call from multiple places unconditionally — `:app`'s `MainActivity` does so from `onResume()` (the sole "check version" trigger point now — see `docs/appupdate-flow.md`) and from `AppUpdateInfoDialog`'s `onSourceModeSaved` callback.

`HostRemoteDataSource`'s verify path (`port`/`verifyPath` constructor params, defaulting to `8080`/`ci-deploy/ci-deploy-version.json`) is parameterized rather than hardcoded, but `:app` is still its only consumer — `:appMouse` doesn't use LAN discovery. Full flow (including how a resolved host feeds both the WebView load and `checkForUpdates`) is in `docs/appupdate-flow.md`.

### Local web console (`:libs:webserver`), configurable port
`HttpWebServerService` (an Android `Service`, bound via `LocalBinder`) serves both `:app`'s web console and `:appQa`'s QA report dashboard on a raw-socket HTTP server (`SimpleHttpServer`). Since a `Service` has no constructor an integrating app can pass args into, the port is set via the companion var `HttpWebServerService.defaultPort` (default `8085`) — **set it before ever starting the service** (`:appQa`'s `MainActivity.onCreate()` sets it to `8086`, first thing, so both apps' consoles can run at once on the same device without colliding). A persisted per-install override also exists (`WebConfig.getPort(default)`/`setPort(value)`, same `PrefsStore`-backed pattern as `web_require_authen`/`web_auth_pin`) but nothing currently exposes UI to change it. Everywhere the port is shown to the user (`WebserverSettingsActivity`'s ADB-forward/URL guide text, `:app`'s `MainActivityWebServer.kt` settings-dialog label, `:appQa`'s status label) reads `defaultPort`/`getPort()` rather than a hardcoded number now — don't reintroduce a literal `8085`.

### Splitting `:app`'s `MainActivity` via Kotlin extension functions
`MainActivity.kt` is large and grows by feature; self-contained slices get split into a sibling file of `internal fun MainActivity.xxx()` extension functions in the same package rather than a separate controller class (see `MainActivityWebServer.kt` for the `:libs:webserver` console integration — service bind/unbind, the console-info label, and the "Web Console Server" section of the settings dialog; `MainActivityBluetooth.kt` for the one-line Bluetooth-test menu launch). Kotlin extensions can't add new instance state, so any stateful fields the split-out functions need (e.g. `webServerService`, `isWebServerBound`, `webServerConnection`) stay as fields on `MainActivity` itself, just widened from `private` to `internal` so the sibling file can reach them. Follow this pattern for the next slice that's genuinely app-specific.

This is a different move from the OTA download/install dialogs (`AppUpdateDownloadFlow`, above) — those went into `:libs:appupdate` proper (a real class, not extension functions) because they're generic enough to be reused by another app, not just split out of one file. Same-app extension file vs. shared-module class is the deciding question when a slice of `MainActivity` grows large: does another app plausibly want this too?
